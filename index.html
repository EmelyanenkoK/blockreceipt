Problem

Individual proof of existence for each document, as it was invented by Manuel Araoz and Esteban Ordano, is very expensive with cost about 3$ per instance at the moment. Moreover, each document requires sepate 240+ bytes transaction stored in blockchain, which restricts the performance of such method to approximately 7 documents per second for Bitcoin blockchain. Thus it fails to meet the requirements for integration into modern document management.

BlockReceipt purpose

Main goal of BlockReceipt protocol & format is introducing a method to use blockchain invariability for securing document management that is affordable, blockchain-friendly and impervious to recovering via brute force attacks.

BlockReceipt's subordinate aim is to provide a convinient and extendable format with attention to simplicity of checking and parsimonious disk usage.

Field descriptions

Each BlockReceipt document (further "receipt") is JSON document that contains all the information needed to verify the data without relying on a trusted third party. Some fields like "v" or "merkleProof" are mandatory, while others like "@format" are optional. All first-level keys in receipt should be treated as case-insensitive. All values and high-order level keys should be treated as case-sensitive.

Mandatory fields
  • "v" - string specifying version of BlockReceipt format
  • "data" - array of elements. First one is mandatory and should be equal sha256 hash of document or null if document hash is concealed.
  • "merkleProof" - unordered Merkle-tree proof for the element (details are further).
  • "*-tx" - transaction that contains unordered Merkle-tree root. Asterisc should be substituted with BTC for bitcoin blockchain, ETH for ethereum blockchain maintained by V.Buterin. Exact rules of embedding described further. More than one *-tx can be presented.
Optional fields
  • "@format" - link to format description
  • Second and further elements in "data". Elements can be arbitrary objects: strings, numbers or collections.
  • "saltedHash" - string with argon2 hashing parameters in accordance to Password Hashing Context specs. hash_len shoulbe set to 32.
  • "merkleRoot" - unordered Merkle-tree root for the element (details are further)
  • "additional_receipts" - list of receipts, composed by the same rules. All receipts should prove the same dochash. This field is added to allows one independently use multiple blockchains and gather gather benefits from all of them. For instance, independently use security of very safe but rare block blockchains like Bitcoin with 10m per block and speed of fast but much less secure blockchains like Ethereum.
  • Unordered merkle tree

    Unordered merkle tree is very similar to simple merkle tree. Usually to get overlying element you need to hash concatenation of two underlying hashes. In unordered merkle tree overlying elements is hash of sum of underlying elements, where elements considered like numbers. This method allows specific changing order of elements without altering merkle-root and merkle-proofs. For instance, tree showed on the picture is insensitive to swap of L1 and L2, as well as swap group (L1,L2) with group (L3,L4).

    As result merkle proofs may not contain order data.

Rules of embedding

Our main requirements for the methods of embedding data into blockchain are: absolute and evident immutability as well as following the established protocols. Currently BlockReceipt supports only two blockchains: bitcoin and ethereum maintained by V.Buterin.

  1. Bitcoin - BlockReceipt uses OP_RETURN transactions with some additional rules:
    • Transaction should contain only one OP_RETURN txout.
    • OP_RETURN should be first opcode in script
    • Second opcode should be OP_PUSH or OP_PUSHDATA* which loads all data after it on stack
    • Only last 32bytes of script should correspond to merkle-root. Data between OP_PUSH and begining of merkle-root is arbitrary and can be used for branding or other purposes.
  2. Ethereum - BlockReceipt uses standard ethereum transactions, last 32bytes of data-field should correspond to merkle-root.
Algorithm of creating receipt

BlockReceipt version 1.

  • Set "v" in the receipt to "1"
  • Create hash of document via sha-256. It should be 32-bytes long.
  • Choose additional info which you want to have incorporated with data. Take into account that this info will be stored in receipt as is.
  • Compose "data" array. Place document hash as hex-encoded 32bytes number into first element of "data" and additional info after it. This "data" array will be called as "fullData" later. If you don't want to show document hash in receipt, for instance you want to give receipt of document without possibility of recovering by brute-force attack, substitute first element of "fullData" with null(take into account that for this case you must use salting). This new array will be called "partialData". When you want to keep document hash in receipt "partialData" is equal to "fullData"
  • Place "partialData" into "data" field of receipt.
  • Stringify "fullData" in such a way as to make all dictionaries (if they are presented in additional data) sorted by key.
  • If you want to salt receipt, use argon2 algorithm with desired parameters (however key-length should be 32bytes) to salt stringified "fullData" and place resulting string(formatted in accordance with Password Hashing Context specs) to "salt" field of receipt. Last 43-letter of this string is salted hash encoded in B64. Decode it to 32bytes hex-string to get "preparedHash" parameter. If you don't want to salt receipt, hash stringified "fullData" with sha256 algo to get "preparedHash".
  • Place "preparedHash" into merkle-tree with hashes for other documents. Create "merkleProof" as ordered array of siblings from the deepest to the second-level from top. Thus first element of array will be hashed with "preparedHash", result of hashing will be hashed with next (second) element of array and so on. Final result of this process will be called further "merkleRoot".
  • Optionally place "merkleRoot" into "merkleRoot" field of receipt.
  • Embed "merkleRoot" int blockchain with accordace to rules of embedding. Result txid place into "*-tx" field of receipt.
Algorithm of checking receipt

BlockReceipt version 1. If any checks failed, receipt doesn't prove document.

  • Check "v" in the receipt. It should be "1".
  • Create hash of document via sha-256. It should be 32-bytes long. Encode it to hex string.
  • If the first element of "data" isn't null compare it with document hash. It should be equal.
  • Compose "fullData" array: get "data" from receipt and substitute first element with document hash.
  • Stringify "fullData" in a such a way to make all dictionaries (if they are presented in additional data) sorted by key.
  • If "salt" field is specified, verify stringified "fullData" via argon2 algorithm and salted parameter. Last 43-letter of this string is salted hash encoded in B64. Decode it to 32bytes hex-string to get "preparedHash" parameter. If salt field doesn't specified, create hash of stringified "fullData" with sha256 algo to get "preparedHash".
  • While "merkleProof" not empty do as follows: pop first element of "merkleProof", sum it with "preparedHash", hash result with sha256 algo and assign to "preparedHash".
  • If "merkleRoot" presents in receipt compare it with "preparedHash", it should be equal.
  • Grab transaction(or transactions) specified in "*-tx". Extract hash in accordance to embedding rules for this blockchain. Compare it with "preparedHash", it should be equal.

  • BlockReceipt vs Chainpoint v.2: these two formats are very similar in essence, however are different in purposes. BlockReceipt is focused on providing convinient and secure method for receipt issuers, thus many unnecessary fields are optional to decrease size and source document hash can be salted to protect data from being recovered. From other hand Chainpoint is focused on providing convinient receipts for checkers. Besides that Chainpoint allows to use different hash algorithms. Both Chainpoint and Blockreceipt allows to anchor data to different blockchains simultaneously, however Chainpoint requires synchronous anchoring.
